# Command and Event Protocol

## Purpose

The protocol is the stable boundary between graph clients and Python. A
WebSocket is the first carrier, but protocol messages should also be usable in
tests, a CLI, or another transport.

Commands flow from a client to Python. Events flow from Python to subscribed
clients. Acknowledgements correlate a command with its acceptance or rejection;
domain events remain the source of truth for resulting state.

## Message envelopes

### Command

```json
{
  "kind": "command",
  "name": "node.spawn",
  "request_id": "req-123",
  "graph_id": "graph-42",
  "expected_revision": 6,
  "payload": {
    "node_id": "multiply-1",
    "node_type": "multiply",
    "position": {"x": 300, "y": 180}
  }
}
```

`request_id` is generated by the client and permits retries and correlation.
`expected_revision` enables optimistic concurrency checks; it may be optional
for commands that do not mutate graph structure.

### Accepted command

```json
{
  "kind": "ack",
  "request_id": "req-123",
  "graph_id": "graph-42",
  "status": "accepted",
  "revision": 7
}
```

Acceptance means the command was applied or queued as defined by that command.
The corresponding domain events carry the resulting state.

### Rejected command

```json
{
  "kind": "ack",
  "request_id": "req-123",
  "graph_id": "graph-42",
  "status": "rejected",
  "error": {
    "code": "pip_not_found",
    "message": "Node multiply-1 has no inbound pip named factor",
    "details": {"node_id": "multiply-1", "pip": "factor"}
  }
}
```

Rejection must leave the graph unchanged.

### Event

```json
{
  "kind": "event",
  "name": "node.output_emitted",
  "event_id": "evt-891",
  "graph_id": "graph-42",
  "sequence": 28,
  "revision": 7,
  "caused_by": "req-123",
  "timestamp": "2026-07-19T12:00:00Z",
  "payload": {
    "node_id": "multiply-1",
    "pip": "result",
    "value": 20
  }
}
```

`caused_by` is present when an event can be traced to a client command. Events
created by timers or background sources may omit it.

## Initial commands

| Command | Purpose |
| --- | --- |
| `graph.create` | Create a new graph session. |
| `graph.attach` | Subscribe to an existing graph and receive a snapshot. |
| `graph.snapshot.get` | Request the current full graph state. |
| `graph.start` | Start automatic queue processing. |
| `graph.stop` | Stop automatic processing without deleting the graph. |
| `graph.step` | Process a defined amount or generation of pending work. |
| `node.spawn` | Create an instance of a registered Python node type. |
| `node.remove` | Remove a node and handle its connected edges. |
| `node.input.set` | Set or emit a value into an inbound pip. |
| `node.execute` | Explicitly queue node execution. |
| `pip.add` | Add an allowed dynamic pip to a node instance. |
| `pip.rename` | Rename a dynamic pip and update edges atomically. |
| `pip.remove` | Remove a dynamic pip under a defined edge policy. |
| `edge.connect` | Connect one outbound pip to one inbound pip. |
| `edge.disconnect` | Remove an existing edge. |
| `node.layout.set` | Persist node position if layout is server-owned. |

Node types may declare fixed pips, dynamic pips, or both. Commands attempting
to edit fixed pips must be rejected.

## Initial events

| Event | Purpose |
| --- | --- |
| `graph.created` | Announces a graph and its initial revision. |
| `graph.snapshot` | Supplies complete nodes, edges, values, and status. |
| `graph.started` / `graph.stopped` | Reports execution-loop state. |
| `graph.queue_changed` | Reports pending work count. |
| `graph.idle` | Reports that no work remains. |
| `node.spawned` / `node.removed` | Updates node projection state. |
| `node.input_updated` | Reports an authoritative inbound value. |
| `node.execution_started` | Marks a node as running. |
| `node.output_emitted` | Reports a value emitted from an outbound pip. |
| `node.execution_failed` | Carries a structured node failure. |
| `pip.added` / `pip.renamed` / `pip.removed` | Updates pip projection state. |
| `edge.connected` / `edge.disconnected` | Updates graph topology. |

## Snapshot shape

```json
{
  "graph_id": "graph-42",
  "revision": 7,
  "last_sequence": 28,
  "status": "running",
  "nodes": [
    {
      "id": "multiply-1",
      "type": "multiply",
      "label": "Multiply",
      "inputs": [
        {"name": "value", "value_type": "number", "value": 10},
        {"name": "multiplier", "value_type": "number", "value": 2}
      ],
      "outputs": [
        {"name": "result", "value_type": "number", "value": 20}
      ],
      "runtime": {"status": "idle", "error": null},
      "layout": {"x": 300, "y": 180}
    }
  ],
  "edges": [
    {
      "id": "edge-8",
      "from": {"node_id": "source-1", "pip": "out"},
      "to": {"node_id": "multiply-1", "pip": "value"}
    }
  ]
}
```

The snapshot contains current state, not a history of execution events.

## Ordering and reconnection

- Sequence numbers are scoped to one graph.
- The server sends graph events to each connection in sequence order.
- The client tracks the last applied sequence.
- A sequence gap causes the client to request a snapshot rather than guess.
- On reconnect, the MVP may always send a fresh snapshot.
- Event replay can be added later without changing envelope fields.

## Idempotency and concurrency

- Reusing a completed `request_id` should return the prior outcome when a
  retry cache is available.
- Creating an existing node ID or edge must not produce an accidental duplicate.
- Structural commands with a stale `expected_revision` should be rejected with
  `revision_conflict` and the current revision.
- A command that affects several resources is one atomic graph operation.

## Value serialization

The MVP accepts JSON values: null, booleans, numbers, strings, arrays, and
objects with string keys. Binary data and arbitrary Python objects require an
explicit codec and must not be silently stringified.

## Transport behavior

One WebSocket can issue commands and receive events. Connections subscribe to
specific graph IDs; the server must not broadcast one graph's values to clients
attached only to another graph.

Malformed JSON or an invalid envelope produces a protocol error without
closing the connection unless the client repeatedly violates the protocol.
